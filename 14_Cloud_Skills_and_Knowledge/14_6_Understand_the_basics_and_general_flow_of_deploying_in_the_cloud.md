## **524. ¬øQu√© es la implementaci√≥n en la nube?**

**Definici√≥n corta:**

La **implementaci√≥n en la nube** (cloud deployment) es el proceso de publicar, poner en marcha y operar una aplicaci√≥n, servicio o infraestructura en un proveedor de nube (AWS, Azure, Google Cloud u otros), o en una combinaci√≥n h√≠brida/privada, de forma automatizada y repetible. A continuaci√≥n te explico qu√© engloba, por qu√© importa, estrategias, pasos concretos y ejemplos pr√°cticos.

### 1. ¬øPor qu√© es importante la implementaci√≥n en la nube?

- **Velocidad:** despliegues r√°pidos y repetibles reducen el time-to-market.
- **Consistencia:** evita diferencias entre entornos (dev/staging/prod).
- **Escalabilidad y resiliencia:** la nube permite escalar y distribuir servicios geogr√°ficamente.
- **Automatizaci√≥n y trazabilidad:** todo queda versionado (c√≥digo + infra), reversible y auditado.
- **Costos y elasticidad:** pagas por lo que usas y escalas seg√∫n demanda.

### 2. Componentes principales de una implementaci√≥n en la nube

1. **C√≥digo / Artefacto:** binario, contenedor Docker, funci√≥n (zip), paquete serverless.
2. **Infraestructura:** VPC, subredes, balanceadores, bases de datos, buckets. (Se crea v√≠a IaC).
3. **Pipeline CI/CD:** construye, prueba, analiza y despliega autom√°ticamente.
4. **Configuraci√≥n / Secrets:** variables de entorno, secretos gestionados (Secrets Manager, Key Vault).
5. **Orquestador / Runtime:** Kubernetes, ECS, App Service, funciones Lambda/Functions.
6. **Monitoreo y Observabilidad:** m√©tricas, logs, tracing, alertas (CloudWatch, Prometheus, DataDog).
7. **Governance:** pol√≠ticas de seguridad, compliance y control de costes.

### 3. Modelos de despliegue en la nube (d√≥nde se implementa)

- **Nube p√∫blica:** AWS, Azure, GCP ‚Äî mayor agilidad.
- **Nube privada:** infra dedicada a una sola organizaci√≥n (on-prem o hosted).
- **H√≠brido:** combinaci√≥n (datos sensibles on-prem + apps en la nube).
- **Multi-cloud:** usar varios proveedores para resiliencia o requisitos espec√≠ficos.

### 4. Estrategias de despliegue (c√≥mo se despliega)

Breve resumen con ejemplos y cu√°ndo usar:

- **Recreate / Replace:** detener versi√≥n antigua y lanzar la nueva. (Simple, riesgo de downtime).
- **Rolling update:** actualizar instancias por lotes para mantener servicio disponible. (K8s, ASG)
- **Blue/Green:** tener dos entornos (blue=prod actual, green=nueva versi√≥n). Cambias tr√°fico al green cuando est√° listo.

  - _Ejemplo:_ cambiar el alias del balanceador o actualizar DNS.

- **Canary:** lanzar la nueva versi√≥n a un peque√±o porcentaje de usuarios; aumentar progresivamente si no hay problemas.

  - _Ejemplo:_ 5% tr√°fico ‚Üí 25% ‚Üí 100%.

- **A/B (feature flags):** exponer funcionalidades solo a ciertos usuarios; ideal para experimentos y rollback instant√°neo.
- **Shadow / Mirroring:** enviar tr√°fico a la nueva versi√≥n sin afectar respuesta real (solo para pruebas de carga y validaci√≥n).

### 5. Pipeline t√≠pico CI/CD (pasos)

1. **Commit** en repo ‚Üí trigger pipeline.
2. **Build**: compilar, empaquetar (jar, docker image).
3. **Tests**: unitarios, integraci√≥n, seguridad (SAST), dependencia scanning.
4. **Push de artefacto** al registry (Docker Hub, ECR, ACR).
5. **Plan de IaC** (`terraform plan` / `arm/bicep` / `cloudformation change set`).
6. **Aprobaci√≥n** (manual si aplica) ‚Üí `apply` infraestructura o release.
7. **Deploy**: actualizar servicio (rolling / canary / blue-green).
8. **Smoke tests & health checks**.
9. **Monitor & promote** a prod o rollback si m√©tricas fallan.
10. **Notificaci√≥n y registro** del despliegue.

### 8. Validaci√≥n post-despliegue y observabilidad

- **Smoke tests**: llamadas simples a endpoints para validar salud.
- **Canary metrics**: latencia, error rate, saturation, business metrics (conversion, checkout).
- **Alertas**: si error rate > X% o latencia subida, activar rollback.
- **Tracing & logs**: correlaci√≥n de request id para depuraci√≥n (OpenTelemetry, X-Ray).
- **SLA/SLO/SLOs**: definir objetivos y medir.

### 9. Seguridad y gobernanza en despliegues

- **Least privilege** para las credenciales del pipeline.
- **Secrets manager** (no variables en repo).
- **Scan de dependencias y SAST** en pipeline.
- **Policy as code**: bloquear infra no permitida (buckets p√∫blicos, reglas inseguras).
- **Auditor√≠a**: registrar qui√©n lanz√≥ qu√© despliegue y cu√°ndo.

### 10. Estrategias de rollback y recuperaci√≥n

- **Rollback r√°pido:** volver a la versi√≥n anterior (tag anterior de imagen).
- **Blue/Green facilita rollback**: cambiar el tr√°fico de vuelta en segundos.
- **Rollback automatizado por m√©tricas**: el pipeline o la plataforma detecta degradaci√≥n y hace rollback autom√°tico.
- **Backups y snapshots**: para bases de datos, tener snapshot previo y plan de restore.

### 11. KPIs y m√©tricas para medir √©xito de la implementaci√≥n

- **Lead time for changes**: tiempo desde commit hasta despliegue en producci√≥n.
- **Deployment frequency**: despliegues por d√≠a/semana.
- **Change failure rate**: % de despliegues que provocan incidentes.
- **MTTR (Mean Time To Recover)**: tiempo medio de restauraci√≥n tras fallo.
- **SLO compliance**: % de tiempo en que se cumplen SLOs.

### 12. Errores frecuentes y c√≥mo evitarlos

- **Desplegar sin pruebas en staging:** siempre validar en entorno identico.
- **No controlar el state de la infra:** usar backend remoto y locking para Terraform.
- **Hardcode de secretos:** usar Secret Manager / Vault.
- **No tener m√©tricas de negocio en canary:** monitorizar solo infra puede esconder regresiones funcionales.
- **Acceso de pipeline con privilegios excesivos:** aplicar least privilege.

### 13. Checklist r√°pido para un despliegue en la nube seguro y confiable

- [ ] C√≥digo empaquetado con versi√≥n (tag).
- [ ] Artefacto publicado en registry seguro.
- [ ] IaC versionado y plan revisado en PR.
- [ ] Pipeline automatizado con tests y SAST.
- [ ] Secrets en gestor seguro.
- [ ] Estrategia de despliegue definida (canary/blue-green).
- [ ] Smoke tests y monitorizaci√≥n configurada.
- [ ] Rollback plan probado.
- [ ] Auditor√≠a y notificaciones (Slack/email) activas.

## **525. Implementaci√≥n de un sitio web en AWS**

- **Sitio est√°tico (recomendado si solo HTML/CSS/JS):** S3 + CloudFront + ACM + Route53. Barato, r√°pido y altamente escalable.
- **Aplicaci√≥n din√°mica (API/backend):** ECR + ECS Fargate (o EKS) + ALB + RDS + Secrets Manager + CloudFront opcional. Escalable y controlado.
- **Infra como C√≥digo:** Terraform (o CloudFormation/CDK) para reproducibilidad.
- **CI/CD:** GitHub Actions o CodePipeline para automatizar builds, despliegues y invalidaciones.

### Requisitos previos

1. Cuenta AWS con permisos suficientes para crear S3, CloudFront, IAM, ECR, ECS, RDS, Route53.
2. AWS CLI configurado (aws configure) o credenciales en CI.
3. (Opcional) Dominio en Route53 o control de DNS en tu proveedor.
4. Terraform (si vas a usar IaC) y GitHub (si vas a usar Actions).

### Parte A ‚Äî **Implementar un sitio est√°tico (S3 + CloudFront)**

#### 1) Idea / arquitectura

- Contenido est√°tico (index.html, CSS, JS, im√°genes) almacenado en **S3**.
- **CloudFront** delante para CDN, HTTPS y protecci√≥n.
- Certificado TLS con **ACM** (en us-east-1 para CloudFront).
- **Route53** apuntando al distribution.
- Habilitar logging, WAF y pol√≠ticas de bloqueo P√∫blico en S3.

#### 2) Pasos manuales (r√°pido)

1. Crear bucket S3:

```bash
aws s3api create-bucket \
  --bucket mi-sitio-ejemplo.com \
  --region us-east-1 \
  --create-bucket-configuration LocationConstraint=us-east-1
```

2. Bloquear acceso p√∫blico y activar versioning/logging:

```bash

  --public-access-block-configuration BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true

aws s3api put-bucket-versioning --bucket mi-sitio-ejemplo.com --versioning-configuration Status=Enabled
```

3. Subir sitio:

```bash

```

4. Crear Origin Access Identity (OAI) para CloudFront y restringir bucket al OAI:

```bash
aws cloudfront create-cloud-front-origin-access-identity --cloud-front-origin-access-identity-config CallerReference=$(date +%s),Comment="OAI for mi-sitio-ejemplo"
# luego ajusta policy del bucket para permitir GetObject al OAI
```

5. Crear certificado ACM en us-east-1 y validar (Route53 validation recomendado).
6. Crear distribuci√≥n CloudFront con origin apuntando al bucket y asociar el certificado TLS.
7. Crear registro DNS en Route53 tipo A (alias) apuntando a la distribuci√≥n CloudFront.

> **Nota:** CloudFront requiere que el certificado ACM est√© en **us-east-1**, aunque el bucket est√© en otra regi√≥n.

### Parte B ‚Äî **Implementar una app din√°mica (ECS Fargate + RDS)**

### Seguridad esencial (aplica a ambos enfoques)

- **Principio de menor privilegio (IAM)**: crea roles espec√≠ficos para CI, tareas ECS y lambdas. Evita usar credenciales ra√≠z.
- **HTTPS/TLS**: ACM para CloudFront/ALB.
- **S3**: bloquear acceso p√∫blico, usar OAI/OAC.
- **Secrets**: AWS Secrets Manager o Parameter Store + encriptaci√≥n KMS. Nunca hardcodees secretos.
